Manual Tester

Static · Design · Exploration

Mission: find defects early and validate the product against real user needs — through static review, designed test cases, and skilled exploration. Shift-left is where you're cheapest to the project: review every story before dev finishes.

ISTQB home turf: static testing (§3) · test analysis & design (§4) · defect management (§5.5) · experience-based techniques · CTAL-TA
11 commands in your kit shift-left, every story every defect → triage

The core loop

One user story, start to finish.

Run this loop on every story. The first two steps happen before development completes — that is the whole point of shift-left.

manual tester — per story
> /qa:static-review "US-123 guest checkout"   # 1 · review the story before dev finishes — find requirement defects now
> /qa:acceptance "US-123"                    # 2 · agree testable acceptance criteria (ATDD / Gherkin) with the team
> /qa:test-cases "US-123"                    # 3 · derive cases with EP, BVA, decision tables (+ CSV export)
> /qa:test-data order                         # 4 · get the data you need to execute
  … execute the scripted cases against the test environment …
> /qa:exploratory "guest checkout"          # 5 · session-based exploration around the scripts (charter, timebox)
> /qa:triage "checkout accepts expired card"  # 6 · every real defect → ISTQB report (severity vs priority)

Scale the technique to the risk

When the feature is bigger than one story.

SituationReach for
Complex business rules / states/qa:test-design <feature> — full conditions + cases, decision tables, state transitions
Many parameters combine (filters, options, configs)/qa:combinatorial <feature> — pairwise instead of exhaustive
A stateful flow (cart, order lifecycle, auth)/qa:mbt <flow> — model it, derive cases by coverage
"Did we miss anything?"/qa:review-coverage <area> — gap analysis vs risk
Unsure which technique applies/qa:istqb-coach "when do I use decision tables?"

Your cadence

When to run what.

Before dev completes

static-review→acceptance — shift-left; cheapest defects to catch

Before execution

test-cases→test-data

After scripted execution

exploratory — one session on the riskiest angle

Every defect found

triage — never report a bug informally

End of sprint

hand stable cases → automate your-cases.csv (scored by ROI)

Review the story before dev finishes — that is the moment a defect costs the project the least.The shift-left rule

Where your work goes

Hand-offs.

Hands off →
Automation Tester — your stable, repetitive manual cases become /qa:automate candidates, scored by ROI.
Test Leader — your execution results and defects feed /qa:status-report.
